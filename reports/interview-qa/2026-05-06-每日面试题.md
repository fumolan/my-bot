## 题目 1：[Java基础/并发] Java 虚拟线程（Virtual Threads / Project Loom）的原理是什么？它与传统线程池相比有哪些优势和适用场景？

**难度**：⭐⭐⭐⭐
**考点来源**：2026-05-03、2026-05-04 报告 — "2026新增考点：虚拟线程替代传统线程池的高并发方案"  
**相关岗位**：Java后端 / 全栈 / 高并发系统开发

### 📖 参考答案

**核心原理：**

Java 虚拟线程（Virtual Threads）是 JDK 21 正式转正的特性，源自 Project Loom。它与传统的**平台线程（Platform Threads）** 有本质区别：

- **平台线程**：是操作系统线程（OS Thread）的 1:1 封装，每个线程占用约 1MB 栈空间，创建/切换开销大
- **虚拟线程**：是 JVM 内部管理的轻量级线程，不与 OS 线程一一对应，多个虚拟线程共享少量**载体线程（Carrier Thread）**

**工作机制（M:N 调度）：**

```
传统线程池（1:1）:
  任务1 → OS线程1
  任务2 → OS线程2     ← 每个任务阻塞 = OS线程阻塞
  任务3 → OS线程3

虚拟线程（M:N）:
  虚拟线程1 ┐
  虚拟线程2 ├→ Carrier线程（ForkJoinPool）
  虚拟线程3 ┘   → 遇到阻塞操作时，虚拟线程被"卸载"（unmount）
                → Carrier线程去执行其他虚拟线程
```

**关键实现细节：**

1. **Continuation 机制**：虚拟线程基于 `Continuation` 实现，当执行到阻塞操作（如 I/O、sleep）时，JVM 自动将虚拟线程从载体线程"卸下"，保存其调用栈状态，载体线程继续执行其他虚拟线程
2. **栈存储**：虚拟线程的栈存储在堆内存中（以 StackChunk 对象形式），可以动态增长和收缩
3. **调度器**：默认使用 `ForkJoinPool` 作为载体线程池

**使用方式：**

```java
// 方式一：直接创建
Thread vThread = Thread.ofVirtual()
    .name("my-virtual-thread")
    .start(() -> {
        // 处理任务
    });

// 方式二：使用虚拟线程 ExecutorService（推荐）
try (ExecutorService executor = Executors.newVirtualThreadPerTaskExecutor()) {
    for (int i = 0; i < 10000; i++) {
        executor.submit(() -> {
            // I/O 密集型任务
            httpClient.send(request);
        });
    }
} // try-with-resources 自动等待所有任务完成并关闭
```

**与传统线程池的对比：**

| 维度 | 传统线程池 | 虚拟线程 |
|------|-----------|---------|
| 线程数量 | 受内存限制（通常几百） | 可轻松创建百万级 |
| 创建开销 | 高（~1ms + 1MB栈） | 极低（~1μs + 数百字节栈） |
| 阻塞代价 | OS线程阻塞，浪费资源 | 自动让出载体线程 |
| 适合场景 | CPU 密集型 | I/O 密集型 |
| 线程复用 | 需要池化复用 | 用完即弃（无需池化） |
| 调试兼容性 | 完全兼容 | 大部分兼容 |

**适用场景与注意事项：**

✅ **最佳场景**：
- Web 服务（每个 HTTP 请求一个虚拟线程）
- 大量 I/O 操作（数据库查询、远程 API 调用）
- 消息消费（每条消息一个虚拟线程处理）

❌ **不适用场景**：
- CPU 密集型计算（应用传统线程池 + ForkJoinPool）
- 需要固定线程数量的场景
- `synchronized` 块中的长时间 I/O 操作（会"钉住"载体线程，JDK 24 已修复）

### 💡 延伸思考

1. **为什么要用 `try-with-resources` 包裹虚拟线程 Executor？** — 因为 `newVirtualThreadPerTaskExecutor()` 创建的 executor 需要在所有任务完成后关闭，try-with-resources 可以确保 `close()` 被调用并等待所有虚拟线程完成。
2. **虚拟线程出现后，线程池设计模式是否过时？** — 对于 I/O 密集型任务是的，"一任务一线程"模式重新被推崇；但 CPU 密集型任务仍需要有限线程池控制并发度。

## 题目 2：[框架/Spring] 请详细解释 Spring Boot 自动配置（Auto-Configuration）的原理，包括 `@EnableAutoConfiguration`、`AutoConfigurationImportSelector` 以及 Spring Boot 3.x 中 `.imports` 文件机制的变化

**难度**：⭐⭐⭐⭐
**考点来源**：2026-05-06 报告 — "Spring Boot 自动配置原理（3.x 迁移到 .imports 文件）"  
**相关岗位**：Java后端 / 微服务开发

### 📖 参考答案

**三层入口：`@SpringBootApplication`**

```java
@SpringBootConfiguration    // 1. 标记为配置类（实质是 @Configuration）
@EnableAutoConfiguration    // 2. 开启自动装配 ← 核心
@ComponentScan              // 3. 组件扫描
public @interface SpringBootApplication {}
```

**`@EnableAutoConfiguration` 内部机制：**

```java
@AutoConfigurationPackage   // 注册自动配置包（用于实体扫描等）
@Import(AutoConfigurationImportSelector.class)  // ← 关键：导入选择器
public @interface EnableAutoConfiguration {}
```

**`AutoConfigurationImportSelector` 工作流程：**

1. 在 Spring 容器刷新时，`ConfigurationClassParser` 解析 `@Import` 注解
2. 触发 `AutoConfigurationImportSelector.selectImports()` 方法
3. 调用 `SpringFactoriesLoader.loadFactoryNames()`（2.x）或读取 `.imports` 文件（3.x）
4. 加载所有候选的自动配置类全限定名
5. 通过**条件注解**过滤，符合条件的配置类才生效

**Spring Boot 2.x vs 3.x 关键变化：**

| 版本 | 配置文件 | 加载方式 |
|------|---------|---------|
| 2.x | `META-INF/spring.factories` | `SpringFactoriesLoader.loadFactoryNames()` |
| 3.x | `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` | `ImportCandidates.load()` |

**2.x 的 spring.factories 示例：**
```properties
# META-INF/spring.factories
org.springframework.boot.autoconfigure.EnableAutoConfiguration=\
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration,\
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
```

**3.x 的 .imports 文件示例：**
```
# META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports
org.springframework.boot.autoconfigure.web.servlet.WebMvcAutoConfiguration
org.springframework.boot.autoconfigure.jdbc.DataSourceAutoConfiguration
```

> **为什么要改？** `.imports` 每行一个类名，简单直接，不与其他 `spring.factories` 条目耦合，解析效率更高。

**条件注解过滤机制（核心筛选器）：**

Spring Boot 提供了一套 `@Conditional` 注解体系来判断某个自动配置类是否生效：

| 条件注解 | 作用 | 示例 |
|---------|------|------|
| `@ConditionalOnClass` | 类路径存在指定类时生效 | `DataSource.class` |
| `@ConditionalOnMissingBean` | 容器中不存在指定 Bean 时生效 | 用户自定义数据源则跳过 |
| `@ConditionalOnProperty` | 配置项满足条件时生效 | `spring.datasource.url` 存在 |
| `@ConditionalOnBean` | 容器中存在指定 Bean 时生效 | `DataSource` Bean 存在后创建 JdbcTemplate |
| `@ConditionalOnMissingClass` | 类路径缺失指定类时生效 | Tomcat 不存在则用 Jetty |

**实际案例理解：**

以 `DataSourceAutoConfiguration` 为例：

```java
@AutoConfiguration
@ConditionalOnClass({ DataSource.class, EmbeddedDatabaseType.class })  // 需要 DataSource 类在路径中
@EnableConfigurationProperties(DataSourceProperties.class)
@Import({ DataSourcePoolMetadataProvidersConfiguration.class, ... })
public class DataSourceAutoConfiguration {
    
    @Configuration
    @ConditionalOnMissingBean(DataSource.class)  // 用户未自定义 DataSource 才自动创建
    static class EmbeddedDatabaseConfiguration {
        // 自动创建 H2 等嵌入式数据源
    }
}
```

**完整流程图：**

```
启动 main() → SpringApplication.run()
    → 创建 Spring 容器
    → refresh() 过程中调用 invokeBeanFactoryPostProcessors()
    → ConfigurationClassParser 解析 @SpringBootApplication
    → 发现 @Import(AutoConfigurationImportSelector.class)
    → AutoConfigurationImportSelector 读取 .imports 文件
    → 获取所有候选自动配置类（100+ 个）
    → 逐一遍历，根据 @Conditional 系列注解过滤
    → 符合条件的配置类被加载，创建对应的 Bean
    → 容器初始化完毕
```

### 💡 延伸思考

1. **如果我想自定义一个 Starter，应该把自动配置类注册在哪里？** — 在 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` 文件中逐行添加你自己的自动配置类，Spring Boot 3.x 会自动发现。
2. **如何禁用某个自动配置？** — 使用 `@SpringBootApplication(exclude = {DataSourceAutoConfiguration.class})` 或在 `application.yml` 中配置 `spring.autoconfigure.exclude`。

**参考来源**：
- Spring Boot 官方文档 — https://docs.spring.io/spring-boot/docs/current/reference/html/auto-configuration.html
- CSDN Spring Boot 高频八股文 — https://blog.csdn.net/Tyyccc_/article/details/160700421

## 题目 3：[数据库/MySQL] 请详细解释 MySQL InnoDB 的 MVCC（多版本并发控制）原理，包括 ReadView、undo log、隐藏列的作用，以及它是如何支持 RC 和 RR 隔离级别的

**难度**：⭐⭐⭐⭐⭐
**考点来源**：2026-05-06、2026-05-05 报告 — "MySQL 事务隔离级别、MVCC、索引优化"  
**相关岗位**：Java后端 / 数据库开发

### 📖 参考答案

**MVCC 是什么：**

MVCC（Multi-Version Concurrency Control）是 InnoDB 实现非锁定一致性读的核心机制。它通过**维护数据的多个版本**，让读操作不被写操作阻塞，写操作也不被读操作阻塞，从而在高并发场景下大幅提升吞吐量。

**四大核心组件：**

1. **隐藏列**
2. **undo log（版本链）**
3. **ReadView**
4. **隔离级别规则**

**隐藏列（每行数据自带）：**

| 隐藏列 | 作用 | 说明 |
|--------|------|------|
| `DB_TRX_ID` | 最近修改该行的事务 ID（6字节） | 记录最后一个插入/更新该行的事务 |
| `DB_ROLL_PTR` | 回滚指针（7字节） | 指向 undo log 中的上一版本 |
| `DB_ROW_ID` | 隐式主键（6字节） | 用户未定义主键时自动生成 |

**undo log 版本链：**

当一条数据被多个事务修改时，undo log 会形成一条版本链：

```
当前记录（聚簇索引中）：
  id=1, name="V3", DB_TRX_ID=300, DB_ROLL_PTR ───┐
                                                   │
undo log 版本链：                                   ▼
  ┌──────────────────────────────────────────────────┐
  │ name="V2", DB_TRX_ID=200, DB_ROLL_PTR ───┐       │
  └──────────────────────────────────────────│───────┘
                                              ▼
  ┌──────────────────────────────────────────────────┐
  │ name="V1", DB_TRX_ID=100, DB_ROLL_PTR = NULL     │
  └──────────────────────────────────────────────────┘
```

**ReadView（快照读的核心）：**

ReadView 是一个"数据快照"，记录当前系统中活跃的读写事务。包含四个关键字段：

| 字段 | 含义 |
|------|------|
| `m_ids` | 生成 ReadView 时，当前系统中活跃的读写事务 ID 集合 |
| `min_trx_id` | `m_ids` 中的最小值 |
| `max_trx_id` | 系统下一个待分配的事务 ID（即当前最大事务 ID + 1） |
| `creator_trx_id` | 生成该 ReadView 的事务 ID |

**可见性判断规则：**

当一条记录的 `DB_TRX_ID` 与 ReadView 比较时：

```
if (DB_TRX_ID == creator_trx_id) {
    // 自己修改的，可见
} else if (DB_TRX_ID < min_trx_id) {
    // 修改该记录的事务在 ReadView 创建前已提交，可见
} else if (DB_TRX_ID >= max_trx_id) {
    // 修改该记录的事务在 ReadView 创建后才开始，不可见
} else if (DB_TRX_ID ∈ m_ids) {
    // 修改该记录的事务在 ReadView 创建时还未提交，不可见
} else {
    // 修改该记录的事务在 ReadView 创建时已提交，可见
}
```

不可见时，通过 `DB_ROLL_PTR` 找到 undo log 中的上一版本，再次判断，直到找到可见版本。

**RC 与 RR 的核心区别：**

| 维度 | READ COMMITTED (RC) | REPEATABLE READ (RR) |
|------|---------------------|---------------------|
| ReadView 生成时机 | **每个 SELECT 语句**都生成新的 ReadView | **仅第一次 SELECT** 生成，事务内复用 |
| 可见性 | 能看到其他事务已提交的最新改动 | 事务过程中看到的始终是事务开始时的快照 |
| 幻读 | 无法解决（当前读会看到新插入的行） | 通过 Next-Key Lock 解决当前读的幻读 |
| 不可重复读 | 存在 | 解决 |

**代码示例——模拟 MVCC 行为：**

```sql
-- 初始数据：id=1, name='Alice'

-- 事务A（RR级别）
START TRANSACTION;
SELECT name FROM users WHERE id = 1;  -- 此时创建 ReadView
-- 结果：'Alice'

-- 事务B（同时执行）
START TRANSACTION;
UPDATE users SET name = 'Bob' WHERE id = 1;
COMMIT;

-- 事务A（继续）
SELECT name FROM users WHERE id = 1;  
-- RC：'Bob'（新 ReadView，能看到事务B已提交）
-- RR：'Alice'（复用原 ReadView，看不到事务B的修改）
```

**当前读 vs 快照读：**

| 类型 | 说明 | SQL 举例 |
|------|------|---------|
| 快照读（Snapshot Read） | 基于 MVCC，不加锁 | 普通 `SELECT` |
| 当前读（Current Read） | 读取最新版本，加锁 | `SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`、`INSERT` |

当前读在 RR 级别下通过 **Next-Key Lock**（记录锁 + 间隙锁）来防止幻读。

### 💡 延伸思考

1. **MVCC 能解决所有并发问题吗？** — 不能。MVCC 只解决**快照读**的并发问题，对于写操作的并发冲突（多个事务同时 UPDATE 同一行），仍需要行锁来保证数据一致性。
2. **为什么 MySQL 默认隔离级别是 RR 而不是 RC？** — 历史原因：MySQL 5.0 以前的 binlog 格式为 STATEMENT，RR 能保证主从复制一致性；虽然现在 ROW 格式 binlog 是默认，但 RR 作为默认惯例延续至今。

- MySQL 官方文档 InnoDB MVCC — https://dev.mysql.com/doc/refman/8.0/en/innodb-multi-versioning.html
- 面渣逆袭 MySQL 篇 — https://javabetter.cn/sidebar/sanfene/mysql.html

## 题目 4：[缓存/Redis] 请详细解释 Redis 缓存雪崩、缓存穿透、缓存击穿的区别，并给出各自的解决方案和实战代码

**难度**：⭐⭐⭐⭐
**考点来源**：2026-05-06、2026-05-05 报告 — "Redis 缓存雪崩/穿透/击穿、分布式锁"  
**相关岗位**：Java后端 / 高并发系统开发

### 📖 参考答案

**三大缓存问题的核心区别：**

```
                    缓存雪崩                       缓存穿透                     缓存击穿
                      │                               │                           │
          大量key同时过期/Redis宕机         查询不存在的数据 (绕过缓存)     热点key过期瞬间
                      │                               │                           │
                      ▼                               ▼                           ▼
            大量请求直接打DB                   请求永远穿透缓存            大量并发请求同一个热点key
                      │                               │                           │
                      ▼                               ▼                           ▼
                  DB被打垮                        无意义的DB压力              DB瞬间高负载
```

---

**一、缓存穿透**

**场景**：查询一个数据库中根本不存在的数据（如 `id=-1`），每次请求都穿过缓存直接打到 DB。

**解决方案**：

**方案1：布隆过滤器（推荐）**

```java
// 使用 Redisson 内置布隆过滤器
RBloomFilter<Long> bloomFilter = redisson.getBloomFilter("product:bloom");
// 初始化：预计 10 万条数据，误判率 0.03
bloomFilter.tryInit(100_000L, 0.03);

// 初始化时，将所有合法 ID 加入布隆过滤器
bloomFilter.add(productId);

// 查询前先判断
public Product getProduct(Long id) {
    if (!bloomFilter.contains(id)) {
        return null; // 直接返回，不查 DB
    }
    // 缓存和 DB 查询逻辑...
}
```

**方案2：缓存空值**

```java
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;
    Product product = redis.get(cacheKey);
    
    if (product != null) {
        // 特殊标记：空对象
        if (product.getId() == -1L) return null;
        return product;
    }
    
    product = db.findById(id);
    if (product == null) {
        // 缓存空值，设置较短过期时间（防占用空间）
        redis.set(cacheKey, new Product(-1L), Duration.ofMinutes(5));
        return null;
    }
    
    redis.set(cacheKey, product, Duration.ofMinutes(30));
    return product;
}
```

---

**二、缓存击穿**

**场景**：某个热点 key 过期瞬间，大量并发请求同时打到 DB。

**解决方案**：

**方案1：互斥锁（分布式锁）**

```java
public Product getProduct(Long id) {
    String cacheKey = "product:" + id;
    Product product = redis.get(cacheKey);
    if (product != null) return product;
    
    // 分布式锁——只让一个线程去查 DB
    String lockKey = "lock:product:" + id;
    RLock lock = redisson.getLock(lockKey);
    
    try {
        // 尝试加锁，等待 5 秒，锁 30 秒后自动释放
        if (lock.tryLock(5, 30, TimeUnit.SECONDS)) {
            // 双重检查：别的线程可能已经重建了缓存
            product = redis.get(cacheKey);
            if (product != null) return product;
            
            // 查询 DB 并回写缓存
            product = db.findById(id);
            if (product != null) {
                redis.set(cacheKey, product, Duration.ofMinutes(30));
            }
            return product;
        }
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    } finally {
        if (lock.isHeldByCurrentThread()) {
            lock.unlock();
        }
    }
    
    // 未获取到锁，短暂等待后重试
    Thread.sleep(50);
    return getProduct(id);  // 递归重试（注意防栈溢出，生产用循环）
}
```

**方案2：逻辑过期（不设 TTL，永不过期）**

```java
// 数据对象封装
class CacheData<T> {
    private T data;
    private LocalDateTime expireTime; // 逻辑过期时间
}

public Product getProduct(Long id) {
    CacheData<Product> cacheData = redis.get("product:" + id);
    if (cacheData == null) return db.findById(id);
    
    // 逻辑过期了
    if (LocalDateTime.now().isAfter(cacheData.getExpireTime())) {
        // 异步重建缓存（获取锁成功后开新线程去更新）
        if (tryLock("lock:product:" + id)) {
            executorService.submit(() -> {
                Product newProduct = db.findById(id);
                redis.set("product:" + id, 
                    new CacheData<>(newProduct, LocalDateTime.now().plusMinutes(30)));
                unlock("lock:product:" + id);
            });
        }
    }
    
    return cacheData.getData(); // 返回旧数据，不阻塞用户
}
```

**三、缓存雪崩**

**场景**：大量缓存 key 在同一时段集体过期，或 Redis 宕机，导致所有请求直达 DB。

**解决方案**：

**方案1：过期时间加随机扰动**

```java
// 不要让所有 key 同时过期！
int baseExpire = 30; // 30分钟
int randomOffset = ThreadLocalRandom.current().nextInt(10); // 0~10分钟随机
redis.set(key, value, Duration.ofMinutes(baseExpire + randomOffset));
```

**方案2：多级缓存（本地缓存 + Redis）**

```java
// Caffeine 本地缓存作为一级缓存，Redis 作为二级缓存
Cache<String, Product> localCache = Caffeine.newBuilder()
    .maximumSize(10000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .build();

public Product getProduct(Long id) {
    String key = "product:" + id;
    
    // L1: 本地缓存
    Product product = localCache.getIfPresent(key);
    if (product != null) return product;
    
    // L2: Redis
    product = redis.get(key);
    if (product != null) {
        localCache.put(key, product); // 回写 L1
        return product;
    }
    
    // L3: 数据库
    product = db.findById(id);
    if (product != null) {
        redis.set(key, product, Duration.ofMinutes(30 + random()));
        localCache.put(key, product);
    }
    return product;
}
```

**方案3：Redis 高可用 + 熔断降级**

```
Redis Sentinel / Cluster → 高可用
Sentinel 熔断 → 限流 → 返回默认值/降级页面
```

| 问题 | 根本原因 | 核心解法 | 辅助手段 |
|------|---------|---------|---------|
| 缓存穿透 | 查不存在的数据 | 布隆过滤器 | 缓存空值 |
| 缓存击穿 | 热点key过期 | 互斥锁/逻辑过期 | 永不过期 |
| 缓存雪崩 | 大量key同时过期或Redis挂了 | 过期时间随机化 | 多级缓存+高可用 |

### 💡 延伸思考

1. **为什么缓存空值方案不是首选？** — 如果恶意攻击者构造大量不存在的 key（如遍历负数ID），内存会被大量空值占满。布隆过滤器空间效率更高，且能直接在内存层拦截。
2. **逻辑过期方案有什么隐患？** — 返回的是"过期的旧数据"，对数据一致性要求极高的场景（如金融账户余额）不适用；适用于"最终一致性"可接受的场景（如商品详情页）。

### 📖 参考答案

**RAG 是什么：**

RAG（Retrieval-Augmented Generation，检索增强生成）是一种让大语言模型（LLM）在回答问题时**先检索外部知识库，再基于检索结果生成回答**的架构。它解决了 LLM 的三大核心问题：

1. **知识滞后**：训练数据有截止日期
2. **幻觉问题**：模型可能编造事实
3. **私有数据**：无法访问企业内部数据

**核心架构（三层）：**

```
┌──────────────┐    ┌──────────────────┐    ┌──────────────┐
│  用户提问     │───▶│  检索层 Retrieval  │───▶│  生成层 LLM   │───▶ 回答
└──────────────┘    │                    │    └──────────────┘
                    │ 1.向量化用户问题    │          │
                    │ 2.向量数据库检索     │          │
                    │ 3.返回Top-K相关文档  │          │
                    └──────────────────┘          │
                                                   ▼
                    ┌──────────────────┐    ┌──────────────┐
                    │  文档/知识库      │    │ Prompt组装    │
                    │  (PDF/Wiki/DB)   │    │ 上下文+问题   │
                    └──────────────────┘    └──────────────┘
```

**RAG 完整工作流程：**

**离线阶段（索引构建）：**

**Java 后端集成 RAG（Spring AI 方案）：**

```java
// Spring AI 提供了良好的 RAG 抽象
@RestController
public class RAGController {
    
    @Autowired
    private VectorStore vectorStore;  // 向量数据库
    
    @Autowired
    private ChatClient chatClient;     // LLM 客户端
    
    @GetMapping("/rag/ask")
    public String ask(@RequestParam String question) {
        // 1. 检索相关文档
        List<Document> docs = vectorStore.similaritySearch(
            SearchRequest.query(question).withTopK(5)
        );
        
        // 2. 拼装上下文
        String context = docs.stream()
            .map(Document::getContent)
            .collect(Collectors.joining("\n\n"));
        
        // 3. 调用 LLM
        return chatClient.prompt()
            .system("基于以下信息回答：\n{context}")
            .user(question)
            .call()
            .content();
    }
}
```

**RAG 评估指标：**

- **命中率（Hit Rate）**：检索结果中包含正确答案的比例
- **MRR（Mean Reciprocal Rank）**：第一个相关文档排名的倒数均值
- **忠实度（Faithfulness）**：生成内容是否严格基于检索到的文档（非幻觉）
- **相关性（Relevance）**：检索结果与问题的语义匹配度
