## 题目 1：[Java基础/JVM] 请详细解释 G1 垃圾收集器的核心原理，包括 Region 分区架构、RSet、SATB 和 Mixed GC 流程，以及与 CMS 的对比

**难度**：⭐⭐⭐⭐⭐
**考点来源**：2026-05-07 早/午间报告 — "JVM：内存模型、GC算法（G1收集器特点高频）、OOM排查、JVM参数调优"  
**相关岗位**：Java后端 / 高并发系统开发 / 基础设施

### 📖 参考答案

**G1 的设计目标：**

G1（Garbage-First）是 JDK 9 起默认的垃圾收集器，设计初衷是替代 CMS，面向**大堆内存（4GB~几百GB）** 且要求**可预测的低停顿时间**（用户可设置期望的停顿时间目标 `-XX:MaxGCPauseMillis`）。

**一、Region 分区架构（G1 的灵魂）**

G1 不再像传统 GC 那样将堆固定划分为连续的 Young/Old 区，而是将整个堆拆分为**大小相等的 Region**（默认约 2048 个，每个 1MB~32MB，通过 `-XX:G1HeapRegionSize` 设置）：

```
传统 GC（Parallel/CMS）：          G1：
   ┌──────────┐                   ┌───┬───┬───┬───┬───┬───┐
   │  Young   │                   │ E │ E │ S │ O │ O │ H │
   ├──────────┤                   ├───┼───┼───┼───┼───┼───┤
   │   Old    │                   │ E │ O │ S │ H │ O │ O │
   └──────────┘                   └───┴───┴───┴───┴───┴───┘
   固定分代                           动态角色的 Region
```

每个 Region 可以是以下四种角色之一：
- **Eden**：新生代对象分配区
- **Survivor**：存活对象区
- **Old**：老年代对象区
- **Humongous**：超大对象区（对象大小 ≥ Region 一半时使用，连续多个 Region）

**二、RSet（Remembered Set）—— 解决跨 Region 引用**

G1 的核心挑战：如果每个 Region 独立回收，怎么知道有没有其他 Region 引用了本 Region 的对象？

```
Region A（Old）               Region B（Young，即将回收）
┌──────────────┐             ┌──────────────┐
│ obj_a ──────────────引用──▶│ obj_b        │  ← 需要知道这个引用！
└──────────────┘             └──────────────┘
```

**RSet 机制：**

```
每个 Region 维护一个 RSet，记录"谁引用了我"：
  Region B 的 RSet = { Region A, offset=128 }  // A 中的某个位置引用了 B

实现：
  ┌─────────────────────────────────────┐
  │ Region A 的 Card Table（卡表）        │
  │  将 Region A 分为 512 字节的 Card   │
  │  ┌───┬───┬───┬───┬───┬───┐         │
  │  │脏卡│   │   │   │   │   │         │ ← 写屏障标记
  │  └───┴───┴───┴───┴───┴───┘         │
  └─────────────────────────────────────┘
           ▼
  Region B 的 RSet 记录：Region A 的 Card 3 引用了我
```

**写屏障（Write Barrier）**：每次引用赋值时，JIT 编译器插入一小段代码，更新卡表标记。这是 G1 能实现"只扫描 RSet 而不扫描全堆"的关键。

**三、G1 垃圾回收全流程**

**3.1 Young GC（新生代回收）**
- **触发条件**：Eden Region 满了
- **过程**：STW，并行复制存活对象到 Survivor/Old Region
- **特点**：只回收 Young Region，通过 RSet 扫描跨代引用

**3.2 并发标记周期（最核心，Mixed GC 的前置阶段）**

这是与 CMS 完全不同的地方，使用 **SATB（Snapshot-At-The-Beginning）** 算法：

| 阶段 | 是否 STW | 工作内容 |
|------|---------|---------|
| **初始标记** | ✅ STW | 标记 GC Roots 直接可达的对象，借用 Young GC 的 STW |
| **并发标记** | ❌ 并发 | 三色标记遍历对象图，与用户线程并发执行 |
| **最终标记** | ✅ STW | 处理并发标记期间的增量变化（SATB 队列处理） |
| **清理** | ✅ STW | 统计 Region 存活对象，按回收价值排序 |

**SATB 算法核心思想：** 在并发标记开始时，对堆做一个"逻辑快照"。并发期间新分配的对象默认当作**活的**（不会被本次回收），而并发期间被删除的引用通过**SATB 队列**记录下来。

**三色标记中的漏标问题与 SATB 解决：**

```
漏标条件（同时满足）：
  1. 并发期间新增了一条从"黑色"到"白色"的引用
  2. 删除了所有从"灰色"到该"白色"的引用

SATB 方案：删除引用时，通过写屏障将被删引用的对象（白色）推入 SATB 队列，
          最终标记阶段重新将其标灰，确保不丢失存活对象。

对比 CMS 的增量更新（Incremental Update）方案：
  增量更新：关注"新增引用"（新增的记录下来重新扫描）
  SATB：关注"删除引用"（被删引用指向的对象要保留）
```

**3.3 Mixed GC（混合回收——G1 的核心特色）**

```
Mixed GC = Young GC + 回收部分 Old Region

策略：按"回收价值/回收成本"排序 Old Region
     → 优先回收垃圾最多的 Region（Garbage-First 名字的由来）
     → 将存活对象拷贝到空闲 Region
     → 在暂停时间目标内回收尽可能多的 Region
```

**3.4 Full GC（兜底）**
- 如果 Mixed GC 来不及回收（并发标记没完成、碎片化严重），退化到 Serial Old 做全堆回收
- **调优目标就是避免 Full GC**

**四、G1 vs CMS 核心对比**

| 维度 | CMS | G1 |
|------|-----|-----|
| 内存布局 | 连续分代 | Region 化（逻辑分代） |
| 并发算法 | 增量更新（Incremental Update） | SATB |
| 碎片化处理 | 不整理，碎片导致 Full GC | Mixed GC 自带拷贝压缩 |
| 停顿预测 | 无 | 有（`-XX:MaxGCPauseMillis`） |
| 大堆表现 | 4GB 以下 OK，越大越差 | 4GB+ 优势明显 |
| 吞吐量 | 较高（轻量） | 略低 10-15%（RSet 维护开销） |
| 适用场景 | 小堆、低延迟、JDK8 | 大堆、低延迟、JDK9+ |

**五、关键 JVM 参数**

```bash
# 启用 G1
-XX:+UseG1GC

# 期望最大停顿时间（软目标，非硬保证，默认 200ms）
-XX:MaxGCPauseMillis=200

# Region 大小（1MB/2MB/4MB/8MB/16MB/32MB）
-XX:G1HeapRegionSize=4m

# 并发标记线程数
-XX:ConcGCThreads=4

# 触发 Mixed GC 的堆占用阈值（默认 45%）
-XX:InitiatingHeapOccupancyPercent=45
```

### 💡 延伸思考

1. **G1 的停顿时间目标 `MaxGCPauseMillis` 为什么是"软目标"？** — 因为 G1 只能尽力满足，如果存活对象太多无法在目标时间内完成拷贝，G1 会适当放宽暂停时间。调优的本质是动态平衡"吞吐量 vs 延迟"。
2. **为什么小堆（<4GB）不推荐用 G1？** — RSet 的维护开销在小堆上占比过高，CMS 或 Parallel GC 开销更低、吞吐量更好。

## 题目 2：[框架/Spring] 请从源码层面解释 Spring IOC 容器的工作原理，包括 BeanDefinition 的加载、Bean 的完整生命周期（12个关键阶段），以及三级缓存如何解决循环依赖

**难度**：⭐⭐⭐⭐⭐
**考点来源**：2026-05-07 早间报告 — "Spring全家桶：IOC/AOP原理、Bean生命周期、Spring Boot自动配置"  
**相关岗位**：Java后端 / 微服务开发

### 📖 参考答案

**一、Spring IOC 的核心本质**

IOC（Inversion of Control）将对象的创建权和依赖维护权从程序员手中转移到 Spring 容器。底层核心是 **工厂模式 + 反射 + 容器抽象**。

```
传统方式：
  UserService service = new UserService(new UserRepository());

Spring IOC：
  @Autowired
  private UserService service;
  // Spring 容器负责：创建 → 注入 → 管理生命周期
```

**两种核心容器接口：**

| 接口 | 特点 | 典型实现 |
|------|------|---------|
| `BeanFactory` | 顶层接口，懒加载（用到才创建） | `DefaultListableBeanFactory` |
| `ApplicationContext` | 继承 BeanFactory，扩展了国际化、事件、资源加载 | `ClassPathXmlApplicationContext`、`AnnotationConfigApplicationContext` |

**二、容器启动流程概览**

```
new AnnotationConfigApplicationContext(AppConfig.class)
  → this()   // 创建 BeanDefinitionReader + Scanner
  → register() // 注册配置类本身
  → refresh() // ← 核心！12 个步骤
      ├── 1. prepareRefresh()         // 环境准备
      ├── 2. obtainFreshBeanFactory() // 创建 BeanFactory
      ├── 3. prepareBeanFactory()     // 配置 BeanFactory
      ├── 4. postProcessBeanFactory() // BeanFactory 后处理（模板方法）
      ├── 5. invokeBeanFactoryPostProcessors() // 执行 BFPP ★
      ├── 6. registerBeanPostProcessors()       // 注册 BPP
      ├── 7. initMessageSource()      // 国际化
      ├── 8. initApplicationEventMulticaster() // 事件广播器
      ├── 9. onRefresh()              // 子类扩展（如启动 WebServer）
      ├── 10. registerListeners()     // 注册监听器
      ├── 11. finishBeanFactoryInitialization() // ★ 实例化所有单例 Bean
      └── 12. finishRefresh()         // 完成刷新，发布事件
```

**三、BeanDefinition 的加载流程**

```java
// 扫描阶段（@ComponentScan）
ClassPathBeanDefinitionScanner
  → 扫描 classpath 下所有 .class 文件
  → 通过 ASM 字节码框架读取类元数据（避免加载类）
  → 判断是否有 @Component 等注解
  → 生成 BeanDefinition 对象（包含：类名、scope、是否懒加载、依赖关系等）
  → 注册到 BeanDefinitionRegistry（本质是一个 ConcurrentHashMap）
```

**四、Bean 的完整生命周期（12 个关键阶段）**

```
┌─────────────────────────────────────────────────────────┐
│  1. 实例化（Instantiation）                               │
│     通过反射调用构造器创建对象实例                              │
│     ├── 优先用 @Autowire 标记的构造器                        │
│     ├── 其次无参构造器                                       │
│     └── 最后有参构造器（参数从容器中找）                         │
├─────────────────────────────────────────────────────────┤
│  2. 属性赋值（Populate Properties）                        │
│     处理 @Autowired、@Value、@Resource 等依赖注入             │
├─────────────────────────────────────────────────────────┤
│  3. Aware 接口回调                                         │
│     BeanNameAware.setBeanName()                            │
│     BeanFactoryAware.setBeanFactory()                       │
│     ApplicationContextAware.setApplicationContext()          │
├─────────────────────────────────────────────────────────┤
│  4. BeanPostProcessor.postProcessBeforeInitialization()   │
│     ★ 重要扩展点：可在此修改 Bean（如 @PostConstruct 支持）      │
├─────────────────────────────────────────────────────────┤
│  5. @PostConstruct 方法                                    │
├─────────────────────────────────────────────────────────┤
│  6. InitializingBean.afterPropertiesSet()                 │
├─────────────────────────────────────────────────────────┤
│  7. init-method（XML 配置或 @Bean(initMethod="xxx")）     │
├─────────────────────────────────────────────────────────┤
│  8. BeanPostProcessor.postProcessAfterInitialization()    │
│     ★ AOP 代理在此阶段生成！                                  │
├─────────────────────────────────────────────────────────┤
│  9. Bean 就绪，可被使用                                      │
├─────────────────────────────────────────────────────────┤
│  10. @PreDestroy 方法                                      │
├─────────────────────────────────────────────────────────┤
│  11. DisposableBean.destroy()                              │
├─────────────────────────────────────────────────────────┤
│  12. destroy-method（自定义销毁方法）                         │
└─────────────────────────────────────────────────────────┘
```

**五、三级缓存解决循环依赖（经典面试题）**

**问题场景：**

```java
@Component
class A {
    @Autowired
    private B b;  // A 依赖 B
}

@Component
class B {
    @Autowired
    private A a;  // B 依赖 A  ← 循环依赖！
}
```

**三级缓存结构：**

```java
// DefaultSingletonBeanRegistry 源码
public class DefaultSingletonBeanRegistry {
    // 一级缓存：存放完全初始化好的 Bean（成品池）
    Map<String, Object> singletonObjects = new ConcurrentHashMap<>();

    // 二级缓存：存放"早期引用"（半成品，已实例化但未完成属性注入）
    Map<String, Object> earlySingletonObjects = new HashMap<>();

    // 三级缓存：存放 ObjectFactory（用于生成早期引用的工厂）
    Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>();
}
```

**解决流程（图解）：**

```
步骤1：创建 A → 实例化 A（构造器） → 把 A 的 ObjectFactory 放入三级缓存
       singletonFactories: { "a": A 的工厂（可提前暴露 A 的原始引用） }

步骤2：A 属性注入 → 发现需要 B → 去容器中获取 B
       → 一级缓存没有 → 开始创建 B

步骤3：创建 B → 实例化 B → 把 B 的 ObjectFactory 放入三级缓存

步骤4：B 属性注入 → 发现需要 A → 去容器中获取 A
       → 一级缓存没有 → 二级缓存没有 → 三级缓存有！
       → 调用 ObjectFactory.getObject() → 获得 A 的早期引用（可能未完成注入）
       → 将 A 的早期引用放入二级缓存，删除三级缓存
       → B 完成属性注入 → 完成初始化 → 放入一级缓存

步骤5：回到 A 的创建流程 → 此时 B 已就绪 → A 完成属性注入
       → A 完成初始化 → 把二级缓存中的半成品 A 升级为成品放入一级缓存
       → 删除二级缓存中的 A

最终状态：
  singletonObjects:      { "a": A@123, "b": B@456 }
  earlySingletonObjects: {}
  singletonFactories:    {}
```

**为什么需要三级缓存（不用两级）？**

关键在于 **AOP 代理**。如果 A 需要被 AOP 代理，`ObjectFactory` 可以在 `getObject()` 时判断是否需要创建代理对象，从而保证注入的是代理对象而非原始对象。二级缓存只能存同一个原始对象，无法解决"A 的原始对象和 A 的代理对象不同"的问题。

**循环依赖解决的前提条件：**
- ✅ 必须是**单例** Bean（prototype 不支持）
- ✅ 必须是**属性注入**或 **setter 注入**（构造器注入无法破解）
- ✅ Spring 默认开启（`allowCircularReferences = true`）

### 💡 延伸思考

1. **为什么构造器注入无法解决循环依赖？** — 构造器调用发生在实例化阶段（生命周期第1步），此时对象还没创建出来，无法提前暴露到缓存。而 setter/属性注入在实例化之后，可以利用三级缓存获得半成品引用。
2. **Spring 6.x（Spring Boot 3.x 对应）为什么默认禁止循环依赖？** — 官方认为循环依赖是一种设计坏味道，默认关闭 `allowCircularReferences`。如果确实存在循环依赖，要么重构代码，要么手动开启。

## 题目 3：[数据库/MySQL] 请详细解释 InnoDB 的行锁机制，包括共享锁/排他锁、记录锁/间隙锁/Next-Key Lock 的区别及适用场景，并分析一个死锁案例

**难度**：⭐⭐⭐⭐
**考点来源**：2026-05-07 早间报告 — "MySQL：锁机制（排他锁/共享锁/间隙锁）"  
**相关岗位**：Java后端 / 数据库开发

### 📖 参考答案

**一、InnoDB 锁的分类全景**

```
InnoDB 锁机制
├── 按操作类型
│   ├── 共享锁（S Lock / 读锁）
│   ├── 排他锁（X Lock / 写锁）
│   └── 意向锁（IS / IX，表级标记）
│
├── 按锁定粒度（行级锁的三种形态）
│   ├── 记录锁（Record Lock）—— 锁住一条记录
│   ├── 间隙锁（Gap Lock）—— 锁住一个区间
│   └── Next-Key Lock（临键锁）—— Record + Gap
│
└── 按实现方式
    ├── 悲观锁（FOR UPDATE / FOR SHARE）
    └── 乐观锁（版本号 / CAS）
```

**二、共享锁与排他锁**

| 锁类型 | SQL 语法 | 兼容性 | 使用场景 |
|--------|---------|--------|---------|
| **共享锁（S）** | `SELECT ... FOR SHARE`（8.0+）或 `LOCK IN SHARE MODE` | S 兼容 S，S 不兼容 X | 读取后不允许其他事务修改，但允许其他事务读 |
| **排他锁（X）** | `SELECT ... FOR UPDATE` | X 不兼容任何锁 | 读取后准备修改，阻止所有其他访问 |

**锁兼容矩阵：**

```
     请求↓   已有→  │  S   │  X
     ──────────────┼─────┼─────
        S          │  ✅  │  ❌
        X          │  ❌  │  ❌
```

**意向锁（Intention Lock）：**

```
意向锁是表级锁，用于协调行锁与表锁：
  IS（意向共享锁）：事务想要在某些行上加 S 锁
  IX（意向排他锁）：事务想要在某些行上加 X 锁

作用：当想加表级 X 锁时，只需检查表上是否有 IX/IS，无需逐行检查。
```

**三、行锁的三种形态（核心面试点）**

**假设表中有索引记录（1, 5, 10, 15），RR 隔离级别下：**

```
数据分布：    [1]       间隔       [5]       间隔       [10]       间隔       [15]
            ↑                                                          ↑
        负无穷                                                      正无穷
```

**3.1 记录锁（Record Lock）**

```sql
-- 精确命中索引
SELECT * FROM t WHERE id = 5 FOR UPDATE;
-- 只锁住 id=5 这一行记录
```

**3.2 间隙锁（Gap Lock）**

```sql
-- 查询不存在的记录
SELECT * FROM t WHERE id = 7 FOR UPDATE;
-- 锁住 (5, 10) 这个间隙，阻止在 5 和 10 之间插入新记录
```

间隙锁是**左开右开**区间，锁住的是**记录之间的空隙**。间隙锁之间不冲突——多个事务可以同时持有同一个间隙的 Gap Lock。

**3.3 Next-Key Lock（临键锁 = Record + Gap）**

```sql
-- MySQL 默认行为的等价操作
SELECT * FROM t WHERE id <= 10 FOR UPDATE;
-- 锁住：(-∞, 1] + (1, 5] + (5, 10] 三段 Next-Key Lock
```

Next-Key Lock 是 InnoDB 在**RR 隔离级别**下默认使用的锁定策略，是记录锁和间隙锁的组合，锁住一个**左开右闭**区间。

```
Next-Key Lock 的区间划分（以现有记录为界）：
  (-∞, 1]  (1, 5]  (5, 10]  (10, 15]  (15, +∞)
```

**不同隔离级别下的锁行为：**

| 隔离级别 | 记录锁 | 间隙锁 | Next-Key Lock | 解决幻读 |
|---------|--------|--------|---------------|---------|
| RC（读已提交） | ✅ | ❌ 关闭 | ❌ | ❌ |
| RR（可重复读） | ✅ | ✅ | ✅（默认） | ✅（当前读场景） |

**加锁的核心前提——走索引：**

```sql
-- 情况1：走索引（索引 id）
SELECT * FROM t WHERE id = 5 FOR UPDATE;
→ 在索引 id=5 的记录上加 X 锁

-- 情况2：不走索引（全表扫描）
SELECT * FROM t WHERE name = 'xxx' FOR UPDATE;  -- name 无索引
→ 所有扫描到的行都会加 Next-Key Lock！极端情况等于锁全表
```

**四、死锁案例分析**

**经典场景：两个事务相反顺序加锁**

```sql
-- 初始数据：id=1, amount=100; id=2, amount=200

-- 事务 A                        -- 事务 B
START TRANSACTION;               START TRANSACTION;

SELECT * FROM account
WHERE id = 1 FOR UPDATE;
-- 锁住 id=1 的 X 锁

                                  SELECT * FROM account
                                  WHERE id = 2 FOR UPDATE;
                                  -- 锁住 id=2 的 X 锁

SELECT * FROM account
WHERE id = 2 FOR UPDATE;
-- 等待事务 B 释放 id=2 的锁...

                                  UPDATE account SET amount = 300
                                  WHERE id = 1;
                                  -- 等待事务 A 释放 id=1 的锁...
                                  -- 💀 死锁！

-- InnoDB 检测到死锁 → 回滚其中一个事务 → 报错：
-- ERROR 1213 (40001): Deadlock found when trying to get lock
```

**死锁预防策略：**

| 策略 | 说明 |
|------|------|
| **统一加锁顺序** | 所有事务按相同顺序访问资源（如都先 id=1 再 id=2） |
| **缩小事务范围** | 锁持有时间越短，死锁概率越低 |
| **使用低隔离级别** | RC 级别关闭间隙锁，减少锁冲突 |
| **乐观锁替代** | 用版本号 `WHERE version = ?` 代替 `FOR UPDATE` |
| **索引优化** | 确保 WHERE 条件走索引，避免锁范围扩大 |

**查看锁信息：**

```sql
-- MySQL 8.0 查看当前锁等待
SELECT * FROM performance_schema.data_locks;

-- 查看死锁日志
SHOW ENGINE INNODB STATUS\G
-- 查看 LATEST DETECTED DEADLOCK 部分
```

### 💡 延伸思考

1. **为什么 RR 级别下使用了间隙锁还会出现幻读？** — 间隙锁只解决"当前读"（`FOR UPDATE`/`UPDATE`/`DELETE`）的幻读问题。普通 `SELECT`（快照读）依靠 MVCC 解决，而当前读通过 Next-Key Lock 阻止新记录插入来防止幻读。
2. **插入操作需要获取什么锁？** — 插入前需要获取**插入意向锁**（Insert Intention Lock），这是间隙锁的一种特殊形式。插入意向锁之间不冲突（同一个间隙可并发插入不同记录），但插入意向锁与间隙锁冲突。

## 题目 4：[缓存/Redis] 请详细解释 Redis Sentinel（哨兵）的高可用机制，包括主观下线/客观下线的判断逻辑、Leader 选举过程以及故障转移的完整流程

**难度**：⭐⭐⭐⭐
**考点来源**：2026-05-07 早间报告 — "Redis 高可用：哨兵 + 集群 Raft 选举、高可用设计"  
**相关岗位**：Java后端 / 基础设施 / 高可用系统设计

### 📖 参考答案

**一、为什么需要 Sentinel？**

传统 Redis 主从架构的三大痛点：

| 痛点 | Sentinel 的解决方案 |
|------|-------------------|
| 主节点单点故障 → 集群写不可用 | 自动故障检测与转移 |
| 手动切换耗时长、容易出错 | 全自动 Failover（通常 10-30 秒完成） |
| 客户端需要感知主节点变更 | 哨兵提供配置中心，返回最新 Master 地址 |

**Sentinel 架构拓扑：**

```
          ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
          │ Sentinel 1   │  │ Sentinel 2   │  │ Sentinel 3   │
          │  (独立进程)   │  │  (独立进程)   │  │  (独立进程)   │
          └──┬───┬───┬───┘  └──┬───┬───┬───┘  └──┬───┬───┬───┘
             │   │   │          │   │   │          │   │   │
          ┌──┘   │   └──────────┼───┘   └──────────┼───┘   │
          ▼      ▼              ▼                  ▼       ▼
     ┌─────────┐     ┌──────────────┐     ┌──────────────┐
     │ Master  │────▶│  Slave 1     │     │  Slave 2     │
     │ (读写)   │     │  (只读/备份)  │     │  (只读/备份)  │
     └─────────┘     └──────────────┘     └──────────────┘
     客户端通过 Sentinel API 获取 Master 地址
```

**二、哨兵的核心功能（四大职责）**

1. **监控（Monitoring）**：周期性 PING，检查 Master 和 Slave 是否存活
2. **通知（Notification）**：通过 Pub/Sub 或 API 通知管理员节点状态变化
3. **自动故障转移（Failover）**：Master 故障时自动选举新 Master
4. **配置中心（Config Provider）**：客户端通过哨兵获取当前 Master 地址

**三、故障检测：主观下线 vs 客观下线**

```
阶段1：主观下线（SDOWN - Subjective Down）
───────────────────────────────────────────
每个 Sentinel 独立判断：
  Sentinel 每 1 秒向 Master 发送 PING
  如果 Master 在 down-after-milliseconds 内未回复
  → 该 Sentinel 标记 Master 为 SDOWN

        Sent-1: Master 失联 → SDOWN ✓
        Sent-2: Master 正常 → 无标记
        Sent-3: Master 失联 → SDOWN ✓

阶段2：客观下线（ODOWN - Objective Down）
───────────────────────────────────────────
当某个 Sentinel 判定 Master 为 SDOWN 后：
  → 向其他 Sentinel 发送询问：IS-MASTER-DOWN-BY-ADDR？
  → 收集投票结果
  → 如果 ≥ quorum（法定人数）个 Sentinel 认为 Master 下线
  → 标记为 ODOWN，触发故障转移

  示例（3 哨兵，quorum=2）：
    Sent-1: SDOWN ✓ → 发询问 → 收到 Sent-3 的确认
    确认数 ≥ 2（quorum=2）→ ODOWN！触发故障转移
```

**四、Leader 选举（类 Raft 算法）**

```
凭什么要选 Leader？
  → 避免多个 Sentinel 同时执行故障转移造成混乱
  → 只有一个 Leader Sentinel 负责执行 Failover

选举过程：
  1. 发现 ODOWN 的 Sentinel 发起选举
  2. 向其他 Sentinel 发送投票请求：
     SENTINEL is-master-down-by-addr <master-ip> <port> <current-epoch> <runid>
  3. 每个 Sentinel 在每个 Epoch（纪元）只能投一票
  4. 获得 > N/2 + 1 票的 Sentinel 成为 Leader（N = Sentinel 总数）
  5. 如果本轮无人胜出，Epoch++，重新投票

  举例（3 个 Sentinel）：
    Sent-1 请求投票，Sent-2 投给它，Sent-3 投给它
    → 得票 3 > 3/2+1 = 2.5 → Sent-1 成为 Leader

为什么建议奇数个 Sentinel？
  ┌──────────┬──────────────────────┐
  │ Sentinel │ 允许故障数 │ 建议   │
  ├──────────┼──────────┼──────────┤
  │    3     │    1     │ ✅ 推荐  │
  │    4     │    1     │ 浪费资源 │
  │    5     │    2     │ ✅ 生产  │
  └──────────┴──────────┴──────────┘
  
  3 个和 4 个 Sentinel 都只能容忍 1 个故障，4 个是浪费。
```

**五、故障转移（Failover）完整流程**

```
  Master 宕机
      │
      ▼
  [1] SDOWN → [2] ODOWN → [3] Leader 选举
                                    │
                                    ▼
                          [4] Leader 执行 Failover
                              │
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
    选择新 Master       提升新 Master       重配置其他 Slave
    │                      │                      │
    ▼                      ▼                      ▼
  规则优先级：          SLAVEOF NO ONE       SLAVEOF <新Master IP>
  ① slave-priority   （从节点变主节点）     （指向新主节点）
     最小的（越低越优）
  ② 复制偏移量最大
     （数据最新）
  ③ runid 字典序最小
     （最终裁决）
```

**选择新 Master 的规则（面试重点）：**

```python
def select_new_master(slaves):
    # 1. 过滤掉不健康的 Slave（SDOWN/ODOWN）
    healthy_slaves = [s for s in slaves if s.is_healthy()]

    # 2. 按 slave-priority 排序（值越小优先级越高，默认 100）
    healthy_slaves.sort(key=lambda s: s.priority)

    # 3. 同优先级，选复制偏移量最大的（数据最新）
    healthy_slaves.sort(key=lambda s: s.replication_offset, reverse=True)

    # 4. 如果还有并列，选 runid 字典序最小的
    return healthy_slaves[0]
```

**七、脑裂问题与解决方案**

```
脑裂场景：
  网络分区导致 Master 被隔离，Sentinel 选出了新 Master
  旧 Master 仍然在接收旧客户端的写入 → 数据不一致！

解决方案（Redis 配置）：
  min-replicas-to-write 1       # 至少 1 个从节点可写
  min-replicas-max-lag 10       # 从节点延迟不超过 10 秒

  如果条件不满足，Master 拒绝写入，避免脑裂期间数据分歧。
```

### 💡 延伸思考

1. **Sentinel 和 Cluster 如何选择？** — Sentinel 解决**高可用**问题（主从切换），适合单主写场景；Cluster 解决**分布式 + 高可用**问题，支持数据分片水平扩展。数据量超过单机内存 → Cluster；只需高可用 → Sentinel。
2. **为什么 Sentinel 的 quorum 和选举多数票可以不同？** — quorum 决定"要不要做 Failover"，选举多数票决定"谁来执行 Failover"。quorum 可以设得较小（快速触发），但选举必须满足多数原则（防止脑裂）。

**三、Tool 定义的结构详解**

以 OpenAI 格式为例，每个 Tool 需要提供精确的结构化描述：

```json
{

  "function": {
    "name": "search_documents",           // 工具名称（唯一标识）

          "type": "integer",
```
